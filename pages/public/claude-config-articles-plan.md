## 系列一：入门理解篇

### 1. CLAUDE.md 完全指南：打造你的 AI 编程规范手册

**一、为什么 CLAUDE.md 如此重要**
- 没有配置的痛点：输出不稳定、风格混乱、质量参差不齐
- CLAUDE.md 的本质：给 Claude Code 的"工作契约"
- 配置文件 vs 每次对话重复说明的效率差异
- 团队协作中的统一规范价值

**二、一份好的 CLAUDE.md 应该包含哪些模块**

1. **核心工作流程**（重要性：高）
   - 定义研究→计划→实现的标准流程
   - 确保 Claude 先理解再动手，避免盲目编码

2. **质量红线/禁止清单**（重要性：高）
   - 明确"绝不允许"的行为边界
   - 提交前的强制检查项

3. **编码标准**（重要性：高）
   - 命名规范、函数长度限制
   - 错误处理模板、空值检查要求

4. **技术栈适配规则**（重要性：中）
   - 前端/后端/数据库的具体规范
   - "遵循项目现有风格"的原则

5. **安全标准**（重要性：高）
   - 输入验证、SQL注入防护
   - 密钥管理、认证授权要求

6. **测试规范**（重要性：中）
   - 覆盖率要求、测试文件组织
   - 测试数据清理原则

7. **Git 与提交规范**（重要性：中）
   - 分支命名、Commit 消息格式
   - 部署检查清单

8. **沟通与问题解决流程**（重要性：低）
   - 禁止的表述方式
   - 不确定时的处理策略

**三、如何开始编写你的第一份 CLAUDE.md**
- 从最痛的问题开始：先解决最影响效率的点
- 渐进式完善：不必一次写完，逐步迭代
- 参考模板与最佳实践

### 2. 三阶段工作流配置：让 Claude 像资深工程师一样思考

**一、为什么需要定义工作流程**
- 没有工作流程的问题：Claude 直接开始写代码，跳过思考
- 典型场景：需求理解偏差、遗漏现有代码、方案不合理
- 工作流程的价值：减少返工、提高一次成功率

**二、三阶段工作流详解**

2.1 研究阶段（RESEARCH）
- 目的：理解现状，避免重复造轮子
- 具体任务：检查现有代码、Glob/Grep 搜索、理解架构、联网搜索
- 配置示例

2.2 计划阶段（PLAN）
- 目的：明确方案，获得用户确认
- 具体任务：列出文件清单、说明方案、识别风险、等待确认
- 配置示例

2.3 实现阶段（IMPLEMENT）
- 目的：高质量交付
- 具体任务：遵循代码风格、完整错误处理、同步写测试、运行检查
- 配置示例

**三、完整配置模板**
- 提供可直接复制使用的完整配置
- 包含注释说明每个部分的作用

**四、进阶配置技巧**
- 4.1 针对复杂任务的扩展（深度思考、先提方案再实现）
- 4.2 针对不同场景的变体（Bug修复、新功能、重构）

**五、常见问题**
- Q: Claude 还是会跳过研究阶段怎么办？
- Q: 如何让 Claude 在计划阶段给出更详细的方案？
- Q: 实现阶段如何确保代码质量？

**六、总结**
- 三阶段工作流的核心价值
- 与其他配置模块的配合
